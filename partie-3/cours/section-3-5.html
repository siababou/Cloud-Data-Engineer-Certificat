Construire des Pipelines de Données en Batch sur Google Cloud

🔄 Résumé : Quand utiliser E-T-L au lieu de E-L-T

Dans la leçon précédente, nous avons vu comment utiliser SQL dans une pipeline E-L-T pour gérer les problèmes de qualité des données. Cependant, il existe des cas où le E-T-L est plus adapté.

📌 Quand E-L-T ne suffit pas :

  • 🌐 Appels à des API externes : Exemple — traduction de texte avec l’API Cloud Translation (ex : espagnol → anglais). Nécessite un code externe à BigQuery.
  • 🧠 Logique métier complexe : Par exemple, l’analyse de séquences d’actions clients dans une fenêtre temporelle est possible en SQL, mais bien plus simple en code.

⚙️ Solution recommandée : E-T-L avec Dataflow

Architecture type :

  1. 📥 Extraction : depuis Pub/Sub, Cloud Storage, Spanner, Cloud SQL...
  2. 🔧 Transformation : avec Dataflow
  3. 📤 Chargement : vers BigQuery

🕒 Cas d’usage pour E-T-L :

  • 📊 Données nécessitant contrôle de qualité ou enrichissement avant chargement
  • 🚀 Transformation trop complexe pour être exprimée en SQL
  • 🌊 Chargement en continu (streaming) requis
  • 🧪 Besoin d’intégration CI/CD avec tests unitaires

🔧 Autres services de transformation de données Google Cloud

  • Dataflow 🧠 — Service serverless basé sur Apache Beam. Gère batch & streaming. Google fournit des templates prêts à l’emploi.
  • Dataproc 🛠️ — Basé sur Apache Hadoop, adapté aux experts Hadoop.
  • Data Fusion 🧩 — Interface graphique intuitive pour construire des pipelines E-T-L déployables sur Dataproc.

✅ Tous ces services permettent de transformer des données avant de les stocker dans un data lake ou un data warehouse pour des analyses avancées.
